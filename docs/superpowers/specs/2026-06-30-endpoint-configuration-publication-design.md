# Endpoint Configuration Publication Design

## Summary

This spec defines the first network-facing design for publishing endpoint
configurations as wallet-signed artifacts.

The current repository already has:
- endpoint manifests;
- immutable configuration snapshots;
- local wallet ownership;
- centralized registry discovery;
- operator-facing endpoint controls.

What is missing is the public trust layer between them:
- the operator needs a way to publish a specific endpoint configuration to the
  network;
- the network needs a stable identifier for that published configuration;
- a remote user needs to verify that the node's live endpoint still matches the
  operator-declared configuration.

This slice introduces that trust layer without requiring a distributed registry
or on-chain settlement first.

## Design Decision

### Selected Direction

Build a `wallet-signed config ledger + registry index` model.

That means:
- endpoint configuration history remains local and append-only;
- selected configurations can be promoted into signed network publications;
- the wallet signs the publication artifact;
- the registry indexes the current published configuration for discovery;
- remote users compare the published `configuration_hash` with a live node proof.

### Why

This keeps trust and discovery separate:
- the wallet proves authorship and ownership intent;
- the registry provides fast discovery and current-state indexing;
- the node still serves live inference and runtime metadata.

This also preserves the operator journey:
- the operator configures an endpoint locally;
- the operator explicitly publishes a configuration;
- the operator can later update local state without silently changing the
  network-visible promise;
- the operator can see whether local state is in sync with the last published
  configuration.

### Rejected Alternatives

#### 1. Registry-Only Configuration Publication

Rejected because the registry should not become the only trust root for endpoint
configuration claims.

That would make future federation or distribution harder and would blur the
boundary between discovery and ownership.

#### 2. Wallet Ledger Without Registry Indexing

Rejected for the first working network slice because it would make discovery and
market visibility too expensive.

Agents and operators need a fast lookup path for "what is the current published
configuration for this endpoint?" without replaying a whole wallet event stream.

## Product Goals

This slice must let the system:
- publish a concrete endpoint configuration as a public network artifact;
- preserve the difference between local configuration and published
  configuration;
- associate the publication with wallet ownership rather than node identity
  alone;
- expose the current published configuration hash through the registry;
- let remote clients verify whether a node still serves the published
  configuration;
- keep configuration publication independent from validation status and rating.

This slice must let the operator:
- inspect local configuration history;
- choose when a local configuration becomes network-visible;
- see whether the local endpoint has drifted from the last published
  configuration;
- publish a replacement configuration without rewriting history.

## Non-Goals

This slice does not:
- implement a distributed registry;
- implement on-chain publication storage;
- define slashing or stake economics;
- replace validation workflow;
- prove live runtime integrity beyond node-reported configuration hash;
- expose every local snapshot to the whole network automatically.

## Core Model

The design has three layers:

1. local configuration snapshot
2. wallet-signed published configuration
3. registry index of current published state

### 1. Local Configuration Snapshot

This already exists in the endpoint domain.

It is:
- append-only;
- generated by endpoint runtime or publication changes;
- local to the operator's node;
- not automatically network-visible.

Each snapshot carries:
- `configuration_hash`;
- `endpoint_id`;
- `bundle_hash`;
- `runtime`;
- `publication`;
- execution config;
- creation timestamp.

### 2. Wallet-Signed Published Configuration

This is the new public trust artifact.

It is produced only when the operator explicitly publishes a configuration.

The network artifact is:
- immutable;
- signed by the endpoint owner's wallet;
- linked to a prior publication when applicable;
- suitable for independent verification by remote users.

Proposed object:

`PublishedEndpointConfiguration`

Fields:
- `schema_version`
- `publication_id`
- `endpoint_id`
- `owner_wallet`
- `node_id`
- `configuration_hash`
- `previous_configuration_hash`
- `bundle_id`
- `bundle_hash`
- `model_class`
- `capabilities`
- `profile`
- `runtime`
- `publication`
- `pricing`
- `validation_requirement`
- `published_at`
- `sequence`
- `status`
- `wallet_signature`

Field meanings:
- `schema_version`: payload version, for example `epcfg.v1`
- `publication_id`: unique record identifier
- `endpoint_id`: stable endpoint identity
- `owner_wallet`: wallet that owns and signs the publication
- `node_id`: node currently binding this endpoint when published
- `configuration_hash`: execution-relevant configuration identifier
- `previous_configuration_hash`: prior published configuration, if any
- `bundle_id`: node-local execution binding reference
- `bundle_hash`: version or artifact reference for execution substrate
- `model_class`: public declared model class
- `capabilities`: public declared capability list
- `profile`: operator-facing public description
- `runtime`: runtime settings relevant to execution behavior
- `publication`: visibility and accessibility settings
- `pricing`: current price contract for this publication
- `validation_requirement`: validation request declaration for this
  publication, not validator outcome
- `published_at`: publication timestamp
- `sequence`: monotonic per-endpoint publication sequence
- `status`: `published`, `superseded`, or `revoked`
- `wallet_signature`: detached or embedded signature over the canonical payload

### 3. Registry Index

The registry remains an index, not the authoritative history store.

For each advertised published endpoint, the registry should expose:
- `endpoint_id`
- `owner_wallet`
- `node_id`
- `current_publication_id`
- `current_configuration_hash`
- `published_at`
- `status`
- `validation_status`
- `price_summary`
- `visibility_summary`
- `bundle_summary`

The registry may also include:
- a compact publication summary for discovery clients;
- a fetch reference or embedded copy of the current published configuration.

## Hashing Rules

### Configuration Hash

`configuration_hash` should identify the operator-declared execution promise,
not the transport metadata around publication.

It should be computed from canonical JSON built from:
- `bundle_hash`
- `model_class`
- `capabilities`
- `runtime`
- `publication`
- `pricing`

It should not include:
- `publication_id`
- `published_at`
- `sequence`
- `status`
- `wallet_signature`
- registry timestamps
- node heartbeat data
- rating
- validator result state

### Why Pricing Is Included

Pricing is part of the public contract.

A remote user comparing published and current node state should be able to
detect if the node changed the commercial promise without publishing a new
configuration.

### Why Profile Is Excluded

`profile` is useful public descriptive metadata, but it is not execution
critical.

Profile changes should be allowed through a new publication record, but they
should not change whether the node is serving the same execution configuration.

## Signature Rules

The wallet signs the canonical JSON payload of the published configuration,
excluding `wallet_signature`.

The signature proves:
- the owning wallet intentionally published this configuration;
- the publication payload has not been altered after signing.

Node identity does not replace wallet signature.

Node identity answers:
- "which node currently serves this endpoint?"

Wallet signature answers:
- "who publicly declared this configuration?"

## Publication Lifecycle

The publication lifecycle is:

1. endpoint changes locally
2. local configuration snapshot is created
3. operator chooses `Publish Configuration`
4. system creates `PublishedEndpointConfiguration`
5. wallet signs it
6. wallet publication ledger stores it
7. registry index moves current pointer to this publication
8. prior publication becomes `superseded`

Optional later state:
- operator revokes current publication;
- publication status becomes `revoked`;
- registry no longer advertises it as current.

## Wallet Publication Ledger

The first milestone does not need a blockchain or external ledger.

It does need a wallet-owned publication journal with stable records that can be:
- exported;
- replayed;
- indexed by sequence;
- verified independently from registry freshness.

Proposed event types:
- `endpoint.configuration_published`
- `endpoint.configuration_superseded`
- `endpoint.configuration_revoked`

The publication ledger should behave similarly to the existing wallet export
streams:
- monotonic ordering;
- replay-safe sequence ids;
- append-only semantics.

## Verification Model For Remote Users

The remote verification path is:

1. discover endpoint through registry
2. read `current_configuration_hash` and `current_publication_id`
3. fetch the signed published configuration
4. verify wallet signature
5. recompute canonical `configuration_hash` from the payload
6. request live node endpoint proof
7. compare live `configuration_hash` to published `configuration_hash`

If the values match, the node is serving the configuration the wallet declared.

If the values differ, the node is out of sync with the published public claim.

## Live Node Proof

The first milestone live node proof can be simple.

The node should expose a public endpoint-level proof contract that returns at
least:
- `endpoint_id`
- `node_id`
- `configuration_hash`
- `bundle_hash`
- current runtime status
- current publication summary

This is not a full attestation system.

It is only a public comparison surface for:
- current live hash;
- current public declaration hash.

## Dashboard UX

The operator dashboard should treat local and published state as separate layers.

For each selected endpoint, the dashboard should show:
- current local `configuration_hash`
- current published `configuration_hash`
- publication status
- sync state:
  - `in sync`
  - `local changes not published`
  - `published configuration not currently served`

New actions:
- `Publish Configuration`
- `Revoke Publication`
- `View Signed Publication`

The current configuration history panel should evolve into:
- local configuration history
- published configuration history

The operator should never have to guess whether a local edit is already visible
to the network.

## API Direction

The first implementation slice should add four new contract areas.

### 1. Endpoint Publication Domain Models

Add models for:
- `PublishedEndpointConfiguration`
- `PublishedEndpointConfigurationRecord`
- `EndpointPublicationVerification`

### 2. Wallet Publication Export

Add export and listing contracts for:
- current publications
- publication event history

Suggested local routes:
- `GET /operators/wallet/endpoints/publications`
- `GET /operators/wallet/endpoints/publications/export`

### 3. Endpoint Publish Actions

Suggested endpoint actions:
- `POST /api/v1/endpoints/{endpoint_id}/publish-configuration`
- `POST /api/v1/endpoints/{endpoint_id}/revoke-publication`
- `GET /api/v1/endpoints/{endpoint_id}/publication`
- `GET /api/v1/endpoints/{endpoint_id}/proof`

### 4. Registry Advertisement Extension

The node advertisement and discovery payloads should grow fields for:
- published endpoint metadata
- current published configuration hash
- publication sync status

This should be additive so current discovery consumers do not break.

## Data Ownership Boundaries

### Endpoint Service

Owns:
- local manifest
- local configuration snapshots
- publication orchestration request

Does not own:
- wallet signature implementation
- registry storage

### Wallet Layer

Owns:
- publication journal
- signature metadata
- export/replay behavior

Does not own:
- runtime control
- registry discovery

### Registry Layer

Owns:
- current public index
- discovery projection

Does not own:
- authoritative publication history
- signature creation

## Risks

### Risk 1: Collapsing Local And Published State

If local configuration edits automatically replace public state, operators lose
control over publication.

Mitigation:
- explicit publish action only;
- explicit sync status in dashboard.

### Risk 2: Hash Drift From Inconsistent Canonicalization

If different surfaces compute hashes differently, remote verification becomes
meaningless.

Mitigation:
- one canonical payload builder in one module;
- tests that recompute hash across publication and proof paths.

### Risk 3: Registry Treated As Source Of Truth

If clients trust the registry without checking signature or live proof, the
system weakens into a discovery-only metadata layer.

Mitigation:
- publish verification flow in docs and API;
- keep signed publication artifact explicit in contracts.

## First Slice Acceptance Criteria

The first milestone is complete when:
- a local endpoint configuration can be published as a wallet-signed artifact;
- a stable `configuration_hash` is computed from execution-relevant fields;
- the wallet publication journal stores append-only publication records;
- the registry advertises the current published configuration hash for an
  endpoint;
- the node exposes a live proof surface for current endpoint configuration;
- the operator dashboard shows local vs published configuration state and sync
  status;
- a remote client can verify that live node hash equals published hash.

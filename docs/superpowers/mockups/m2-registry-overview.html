Три пути для первого релиза

A. Только каталог нод

Реестр хранит только базовую информацию о ноде без bundle-level деталей.

быстро мало схем слабый discovery
  • Подходит для раннего smoke-stage.
  • Плохо отвечает на вопрос, какой endpoint реально доступен агенту.
  • Придётся быстро переделывать под bundle-level модель.
Рекомендация

B. Node Registry + Bundle Discovery

Реестр принимает heartbeat от ноды и хранит одновременно node-level и bundle-level capabilities.

правильный MVP достаточно конкретно без федерации
  • Агент видит ресурсы, провайдеров, цены, рейтинг и endpoints в одном ответе.
  • Не тащит распределенность раньше времени.
  • Хорошо совместим с будущими wallet/rating слоями.

C. Сразу market-ready registry

С первого дня включаем settlement, рейтинг, dispute-поля и почти финальную сетевую модель.

богатый контракт высокий риск медленный старт
  • Архитектурно красиво, но тяжело двигать итерациями.
  • Смешивает discovery и market concerns слишком рано.
  • Высокий шанс затормозить сетевой запуск.

Рекомендуемый контракт

Первый shipping-срез: один registry service, куда гипервизоры регистрируются сами, публикуют heartbeat и bundle inventory, а агент делает фильтрованный discovery-запрос.
1. Node Registration `node_id`, operator, base URL, heartbeat TTL, totals по CPU/RAM/VRAM, `can_host_custom_model`.
2. Bundle Publication Для каждой schedulable bundle: `workload_type`, `provider_type`, `model_id`, endpoint, launch mode, enabled, pricing.
3. Discovery API Фильтры по `workload_type`, `provider_type`, `model_id`, `can_host_custom_model`, max price, min rating.
4. Freshness Rules Если heartbeat устарел, нода скрывается или деградирует в discovery.

Почему не брать distributed сейчас

Федерация остаётся целевой архитектурой, но для ближайшего релиза это лишняя сложность: сначала нужно стабилизировать metadata contract, heartbeat semantics и discovery ranking.
  • Сначала фиксируем один source of truth для `node + bundle` публикации.
  • Затем добавляем pricing/rating как стабильные поля того же контракта.
  • Только после этого имеет смысл подписывать объявления и реплицировать их между registry peers.

Если этот вариант выглядит правильно, следующий шаг — зафиксировать его в spec/plan и реализовать первый технический срез: in-memory registry service + node advertisement endpoint + discovery endpoint.
